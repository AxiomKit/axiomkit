## What is MCP?

The Model Context Protocol (MCP) is an open protocol that allows AI agents to discover and use external capabilities in a consistent way. MCP servers expose capabilities like tools, prompts, and resources over a transport (stdio or SSE). Clients connect to an MCP server, enumerate its capabilities, and invoke tools via a schema-validated interface.

In practice, MCP lets you:
- Define command-like tools with structured input/output
- Host them behind a standard server interface
- Connect multiple agents/clients to reuse those tools

## AxiomKit + MCP

AxiomKit integrates MCP in two ways:
- Provider for agents: `createMcpProvider(servers)` connects your agent to one or more MCP servers and exposes their tools as first-class agent actions.
- Server helper: `createMcpServer(options)` lets you build an MCP server quickly with type-safe schemas (Zod v4 supported).

### Exports
- `createMcpProvider(servers)` from `@axiomkit/mcp`
- `createMcpServer({ name, version })` from `@axiomkit/mcp`
- Types and schemas: `McpServerConfig`, `McpClientConfig`, etc.

## Quick Start: Connect an Agent to an MCP Server

```typescript
import { createAgent } from "@axiomkit/core";
import { createMcpProvider } from "@axiomkit/mcp";
import { groq } from "@ai-sdk/groq";
import path from "node:path";

const agent = createAgent({
  model: groq("gemma2-9b-it"),
  providers: [
    createMcpProvider([
      {
        id: "simple-server",
        name: "Simple MCP Server",
        transport: {
          type: "stdio",
          command: "npx",
          args: ["-y", "tsx", path.join(__dirname, "mcp-simple-server.ts")],
        },
      },
    ]),
  ],
});

await agent.start();
```

**Note:** Using `npx -y tsx` ensures `tsx` is found even if not globally installed. The `-y` flag auto-accepts the package installation prompt.

## Build a Simple MCP Server

Create a server using `createMcpServer`. The example below exposes three tools: `get_info`, `echo`, and `add`.

```typescript
// mcp-simple-server.ts
import { createMcpServer } from "@axiomkit/mcp";
import * as z from "zod";

const server = createMcpServer({
  name: "Simple MCP Test Server",
  version: "1.0.0",
});

// No-arg tool
server.tool("get_info", {}, async () => {
  return {
    content: [
      { type: "text", text: "Hello from AxiomKit MCP! This server is working correctly." },
    ],
  };
});

// Echo tool with schema
server.tool(
  "echo",
  { message: z.string().describe("Message to echo back") },
  async ({ message }) => {
    return { content: [{ type: "text", text: `Echo: ${message}` }] };
  }
);

// Simple math tool
server.tool(
  "add",
  {
    a: z.number().describe("First number"),
    b: z.number().describe("Second number"),
  },
  async ({ a, b }) => {
    const result = a + b;
    return { content: [{ type: "text", text: `${a} + ${b} = ${result}` }] };
  }
);

async function main() {
  console.error("Simple MCP Test Server starting...");
  await server.connect(); // stdio transport by default
}

main().catch((error) => {
  console.error("Fatal error in main():", error);
  process.exit(1);
});
```

### Run the Server (dev)

```bash
npx tsx mcp-simple-server.ts
```

The server writes protocol traffic on stdout and logs on stderr, as required by MCP. Keep it running to allow clients to connect.

## Connect Your Agent to the Simple Server

Point the provider to the server entry using stdio. Use `npx` to ensure `tsx` is found:

```typescript
import { createAgent } from "@axiomkit/core";
import { createMcpProvider } from "@axiomkit/mcp";
import { groq } from "@ai-sdk/groq";
import path from "node:path";

const agent = createAgent({
  model: groq("gemma2-9b-it"),
  providers: [
    createMcpProvider([
      {
        id: "simple-server",
        name: "Simple MCP Server",
        transport: {
          type: "stdio",
          command: "npx",
          args: ["-y", "tsx", path.join(__dirname, "mcp-simple-server.ts")],
        },
      },
    ]),
  ],
});

await agent.start();
```

When your agent runs, it can list MCP tools and invoke them as actions. The `createMcpProvider` handles connection lifecycle, retries, and safe execution.

### Alternative: Using Local tsx Binary

If `tsx` is installed locally in your project, you can use the full path:

```typescript
import { createAgent } from "@axiomkit/core";
import { createMcpProvider } from "@axiomkit/mcp";
import { groq } from "@ai-sdk/groq";
import path from "node:path";

// Get the path to local tsx binary
const tsxPath = path.join(process.cwd(), "node_modules", ".bin", "tsx");

const agent = createAgent({
  model: groq("gemma2-9b-it"),
  providers: [
    createMcpProvider([
      {
        id: "simple-server",
        name: "Simple MCP Server",
        transport: {
          type: "stdio",
          command: tsxPath,
          args: [path.join(__dirname, "mcp-simple-server.ts")],
        },
      },
    ]),
  ],
});

await agent.start();
```

## Implementation Notes

- Transports: `stdio` is recommended for local development; `sse` is available for HTTP-based servers.
- Schemas: Use Zod v4. The server adapts schemas for MCP SDK compatibility.
- Logging: MCP servers must keep stdout clean for protocol messages; logs are routed to stderr.

## Troubleshooting

### "Executable not found in $PATH: tsx"

This error occurs when `tsx` is not available in the system PATH. Solutions:

1. **Use `npx` (Recommended):** Change your transport config to use `npx`:
   ```typescript
   transport: {
     type: "stdio",
     command: "npx",
     args: ["-y", "tsx", path.join(__dirname, "mcp-simple-server.ts")],
   }
   ```

2. **Install tsx globally:**
   ```bash
   npm install -g tsx
   # or
   pnpm add -g tsx
   ```

3. **Use local binary path:** Reference the local `node_modules/.bin/tsx` directly:
   ```typescript
   const tsxPath = path.join(process.cwd(), "node_modules", ".bin", "tsx");
   transport: {
     type: "stdio",
     command: tsxPath,
     args: [path.join(__dirname, "mcp-simple-server.ts")],
   }
   ```

### Other Common Issues

- **No tools visible:** Ensure your server is running and the transport command/path is correct.
- **Schema errors:** Verify Zod types and that arguments match tool schemas.
- **Stdio hangs:** Confirm the command is resolvable and that the process remains running (no premature exit).
- **Connection retries:** Check server logs (stderr) for startup errors. The provider will retry up to 3 times by default.
