# Agent Architecture

## Overview

Axiomkit's architecture is designed around the principle of **composable autonomy**. Each agent is built from modular components that work together to create intelligent, adaptable systems. This architecture enables agents to:

- **Scale from simple to complex** - Start with basic functionality and add capabilities as needed
- **Maintain state across sessions** - Remember context and learn from interactions
- **Adapt to changing requirements** - Modify behavior through configuration and extensions
- **Integrate with external systems** - Connect to APIs, databases, and services seamlessly

## Core Components

### 🧠 Agent Runtime

The **Agent Runtime** is the central orchestrator that coordinates all other components. It manages:

- **Event Processing** - Handles incoming messages, API calls, and triggers
- **Action Dispatch** - Routes requests to appropriate actions and extensions
- **Memory Management** - Coordinates short-term and long-term memory operations
- **Context Switching** - Manages different conversation contexts and domains

```typescript
import { createAgent } from "@axiomkit/core";

const agent = createAgent({
  model: groq("deepseek-r1-distill-llama-70b"),
  extensions: [/* your extensions */],
  memory: {
    provider: "mongodb", // or "in-memory", "supabase", etc.
    config: { /* memory configuration */ }
  }
});
```

### 🎯 Contexts

**Contexts** are isolated containers for different types of interactions. They provide:

- **Conversation Isolation** - Separate memory and state for different users or topics
- **Domain Specialization** - Tailored behavior for specific use cases
- **State Management** - Persistent storage of relevant information

```typescript
// Create a context for customer support
const supportContext = {
  type: "support",
  instructions: [
    "You are a customer support agent",
    "Always be helpful and professional",
    "Escalate complex issues to human agents"
  ],
  memory: {
    customerHistory: true,
    ticketTracking: true
  }
};

// Use the context in agent interactions
const result = await agent.run({
  context: supportContext,
  args: { message: "I need help with my order" }
});
```

### ⚡ Actions

**Actions** are the building blocks of agent behavior. They define what an agent can do:

- **Function Declarations** - Describe capabilities to the LLM
- **Implementation** - Actual code that executes the action
- **Type Safety** - Full TypeScript support for inputs and outputs

```typescript
import { createAction } from "@axiomkit/core";

// Define a simple action
const sendEmail = createAction({
  name: "send_email",
  description: "Send an email to a specified recipient",
  parameters: {
    to: { type: "string", description: "Recipient email address" },
    subject: { type: "string", description: "Email subject" },
    body: { type: "string", description: "Email body content" }
  },
  execute: async ({ to, subject, body }) => {
    // Implementation here
    await emailService.send({ to, subject, body });
    return { success: true, messageId: "msg_123" };
  }
});
```

### 🧠 Memory System

The **Memory System** provides structured persistence across sessions:

- **Short-term Memory** - Recent interactions and current context
- **Long-term Memory** - Persistent knowledge and learned patterns
- **Semantic Search** - Find relevant information using natural language queries
- **Memory Summarization** - Condense long conversations into key insights

```typescript
// Query memory for relevant information
const relevantInfo = await agent.memory.query({
  question: "What did we discuss about Project X last week?",
  context: "user_123",
  limit: 5
});

// Store new information
await agent.memory.store({
  content: "User requested feature Y for Project X",
  metadata: {
    userId: "user_123",
    project: "Project X",
    timestamp: new Date()
  }
});
```

### 🔌 Extensions

**Extensions** bundle related functionality into reusable modules:

- **Platform Integrations** - Discord, Telegram, Slack, etc.
- **Service Connectors** - Database, API, and external service integrations
- **Custom Logic** - Domain-specific behaviors and workflows

```typescript
import { createTelegramExtension } from "@axiomkit/telegram";

// Create a Telegram extension
const telegramExt = createTelegramExtension({
  botToken: process.env.TELEGRAM_BOT_TOKEN,
  commands: {
    "/start": "Welcome message",
    "/help": "Show available commands"
  },
  actions: [sendEmail, createTicket] // Available actions
});

// Add to agent
const agent = createAgent({
  model: groq("deepseek-r1-distill-llama-70b"),
  extensions: [telegramExt]
});
```

### 🤖 LLM Providers

**LLM Providers** abstract interaction with language models:

- **Model Agnostic** - Switch between providers without code changes
- **Unified Interface** - Consistent API across different models
- **Provider Extensions** - Add custom providers easily

```typescript
import { groq } from "@ai-sdk/groq";
import { openai } from "@ai-sdk/openai";
import { anthropic } from "@ai-sdk/anthropic";

// Use any supported provider
const agent1 = createAgent({ model: groq("deepseek-r1-distill-llama-70b") });
const agent2 = createAgent({ model: openai("gpt-4") });
const agent3 = createAgent({ model: anthropic("claude-3-sonnet") });
```

## Agent Lifecycle

### 1. Initialization
```typescript
// Agent is created with configuration
const agent = createAgent(config);

// Extensions and memory are initialized
await agent.start();
```

### 2. Event Processing
```typescript
// Agent receives input (message, API call, etc.)
const result = await agent.run({
  context: contextConfig,
  args: { message: "Hello, can you help me?" }
});
```

### 3. Reasoning & Planning
- LLM analyzes the input and context
- Determines appropriate actions to take
- Plans the execution sequence

### 4. Action Execution
- Actions are executed with proper parameters
- Results are captured and stored
- Memory is updated with new information

### 5. Response Generation
- Final response is generated based on action results
- Context is updated for future interactions
- Memory is persisted for long-term learning

## Memory Architecture

### Memory Types

| Type | Purpose | Persistence | Use Case |
|------|---------|-------------|----------|
| **Working Memory** | Current session state | Temporary | Active conversation context |
| **Episodic Memory** | Past interactions | Long-term | Learning from experience |
| **Semantic Memory** | Knowledge and facts | Long-term | Domain expertise |
| **Procedural Memory** | Skills and procedures | Long-term | Action patterns |

### Memory Providers

- **In-Memory** - Fast, temporary storage for development
- **MongoDB** - Scalable, persistent storage for production
- **Supabase** - PostgreSQL-based storage with real-time features
- **Custom** - Implement your own storage backend

## Extension Architecture

### Built-in Extensions

| Extension | Purpose | Use Cases |
|-----------|---------|-----------|
| **@axiomkit/telegram** | Telegram bot integration | Customer support, notifications |
| **@axiomkit/discord** | Discord bot integration | Gaming, community management |

### Creating Custom Extensions

```typescript
import { createExtension } from "@axiomkit/core";

const customExtension = createExtension({
  name: "my-custom-extension",
  version: "1.0.0",
  
  // Define actions
  actions: [
    customAction1,
    customAction2
  ],
  
  // Define contexts
  contexts: [
    customContext1,
    customContext2
  ],
  
  // Lifecycle hooks
  onStart: async () => {
    console.log("Extension started");
  },
  
  onStop: async () => {
    console.log("Extension stopped");
  }
});
```

## Best Practices

### 1. Context Design
- **Keep contexts focused** - Each context should have a clear, specific purpose
- **Use descriptive names** - Make context types self-documenting
- **Include relevant instructions** - Provide clear guidance for the LLM

### 2. Action Design
- **Single responsibility** - Each action should do one thing well
- **Clear parameters** - Use descriptive parameter names and types
- **Error handling** - Always handle potential failures gracefully
- **Type safety** - Leverage TypeScript for compile-time safety

### 3. Memory Management
- **Relevant storage** - Only store information that will be useful later
- **Efficient queries** - Use semantic search to find relevant information
- **Regular cleanup** - Archive or delete old, irrelevant data

### 4. Extension Development
- **Modular design** - Keep extensions focused and reusable
- **Version management** - Use semantic versioning for your extensions
- **Documentation** - Provide clear documentation for your extension's capabilities

## Performance Considerations

### Memory Optimization
- **Selective storage** - Only store essential information
- **Efficient queries** - Use indexes and optimized search patterns
- **Regular cleanup** - Implement data retention policies

### Response Time
- **Action optimization** - Keep actions fast and efficient
- **Parallel execution** - Execute independent actions concurrently
- **Caching** - Cache frequently accessed data

### Scalability
- **Horizontal scaling** - Distribute agents across multiple instances
- **Database optimization** - Use appropriate indexes and query patterns
- **Resource management** - Monitor and optimize resource usage
